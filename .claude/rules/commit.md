# コミットルール

[Conventional Commits](https://www.conventionalcommits.org/ja/v1.0.0/) に従う。

## 心構え

コミットは「変更を取り込んでもらうための説得」である。なぜこの変更が必要なのか、何が改善されるのかをレビュアーに伝え、マージする価値があると納得してもらうために書く。この考え方は [Mercurial ContributingChanges](https://wiki.mercurial-scm.org/ContributingChanges) に倣う。

## コミットの粒度

1つのコミットは1つの明確な目的だけを持つ。subject を箇条書きで書きたくなったら分割すべきサインである。

- 機能的な変更、フォーマット調整、コード移動、タイポ修正などを混在させない
- 異なる type に属する変更は別のコミットに分ける
- 大きな変更は小さなコミットの連続に分割する。各コミットは実装プロセスの1ステップを表す
- どのコミットの時点でもコードは動作する状態を保つ（問題の原因となったコミットを特定しやすくなる）
- テストの変更は、対応する実装と同じコミットに含める
- コミットが小さいほどレビューしやすく、採用されやすく、バグも少ない

## コミットメッセージのフォーマット

```
<type>: <subject>

<body>（任意）
```

- `type`: feat, fix, chore, docs, style, refactor, perf, test, ci, build
- `scope`: 原則省略する。必要な場合のみ `<type>(<scope>): <subject>` の形式で付ける
- `subject`: 日本語・命令形で書く（「〜する」）
- `body`: 変更の動機・正当性・代替案を説明する

## subject の書き方

subject は命令形で書く。コードベースに動作変更を指示するように記述する。
「このコミットを適用すると〜する」と読めるようにする。

問題ではなく修正内容（何をするか）を要約する。

- OK: `ログイン機能を追加する`、`パースエラーを修正する`
- NG: `ログイン機能を追加した`、`パースエラーの修正`（過去形・体言止めは使わない）
- NG: `ログイン時にエラーが出る`（問題の記述であり、修正内容の要約ではない）

## body の書き方

body には以下を含める：

1. **問題**: 変更しない場合のコードの何が問題なのかを説明する
   - 「Xを試みるとYになるが、Zであるべき」のように具体的に書く
   - 現在形で書く（「〜している」ではなく「〜する」）
   - 変更を加えていないコードの状態として記述する
   - ただし、一連のコミットの流れから当然導かれる問題は省略してよい（例: ライブラリ更新コミットの直後にフォーマット適用コミットが続く場合、「フォーマットが未適用」は自明なので問題として書かない）
2. **解決**: 変更によって問題がどう解決されるかを正当化する。なぜこの変更による結果が優れているかを説明する
   - 問題を省略した場合は、何を実行するかだけを簡潔に書く
3. **代替案**（該当する場合）: 検討したが採用しなかった代替の解決策があれば記載する

議論の要点は body 内に要約した上で、関連する issue・Slack スレッド・ドキュメントなどの参照 URL を積極的に貼る。URL だけを貼って説明を省略しない。要約と URL の両方があることで、コンテキストの理解と原典へのアクセスを両立させる。

## 例

```
feat: ログイン機能を追加する
fix: レスポンスのパース失敗を修正する
chore: dependencyを更新する
refactor: コンポーネント構造を整理する
```

bodyを含む例（問題・解決・代替案の構造）：

```
fix: タイムアウト時のリトライ処理を追加する

APIへのリクエストがタイムアウトするとリトライされず、ユーザーにエラーが表示される。
一時的なネットワーク障害でも操作が失敗するが、リトライすれば回復できるはずである。

指数バックオフ付きのリトライを導入し、一時的な障害への耐性を高める。
3回までリトライすることで、大半の一時障害を透過的に回復できる。
```

```
perf: reachable_roots の探索アルゴリズムを改善する

リビジョン数が多いリポジトリで reachable_roots の実行に約2秒かかる。
10万リビジョン規模では体感できる遅延となり、操作のたびに待たされる。

探索済みノードのキャッシュを導入し、重複探索を排除する。
10万リビジョンのリポジトリで約2秒から約0.3秒に短縮される。
BFS への切り替えも検討したが、メモリ使用量が増大するため、キャッシュ方式を採用する。
```

```
refactor: PathUtils を core モジュールに移動する

PathUtils は現在 cli モジュールに属するが、パス操作ロジックは CLI に依存しない汎用的な処理である。
今後 FUSE VFS の実装で同じユーティリティを利用する予定があり、cli への依存を避けたい。

core モジュールに移動し、cli と今後の FUSE VFS の両方から参照できるようにする。
公開インターフェースは変更しない。
```

## クイックリファレンス

- subject は日本語・命令形で書く（「〜する」で終わる）
- type と scope は英語（小文字）
- subject の末尾にピリオドをつけない
- subject は72文字以内、body は1行100文字以内を目安にする
- Breaking Change がある場合は `!` を付ける（例: `feat!: 認証方式を変更する`）
- body では問題・解決・代替案の構造で書く
