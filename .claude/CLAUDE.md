# CLAUDE.md

## ディレクトリ

- 一時作業ディレクトリ: `~/.cache/claude-scratch/`

## スタイル

- Claude が生成するテキスト（応答・ドキュメント）は簡潔かつ明瞭に書く
- 装飾目的での絵文字は避ける
- 事実と推測を区別する
  - 裏付けのある情報（公式ドキュメント、ソースコード、実測値）と推論・一般論を混ぜて断言しない
  - 推測には「〜と考えられる」「根拠は未確認だが」等の留保を付ける
  - ベストプラクティスや慣習を根拠にする場合は、出典を示せるか自問する

## 日本語ドキュメントの文体

README / CONTRIBUTING など人間向けの解説ドキュメントを書くときは次のスタイルに従う。CLAUDE.md / `~/.claude/rules/` 配下のルールファイル / SKILL.md のような LLM 向けルールファイル（箇条書き命令形が自然な文書）は対象外。

- 文書全体としてはですます調か「である」調のどちらかを選んで統一する（混在しない）
- 個別 CLAUDE.md で別途指定がない限り、基本ですます調
- 同じ活用の語尾が反復してくどく単調に感じる箇所は、体言止め（名詞で文を終える: 「〜の JSON 配列。」「〜の composite action。」）を混ぜて整える
  - 「します・なります・されます」のように活用が異なれば語尾の音にバリエーションが出るので問題なし。「します・します・します」のように同じ活用が反復するときが対象
  - 「〜する」のような動詞終止形は「である」調側のニュアンスになるので、ですます調文書では使わず、体言止めで対応する
- 並列に並ぶ列挙項目（動作リスト、手順、bullet など）はですますが連続しても並列感が出て自然なことが多いので、無理に変えない
- 機械的に直すと逆に不自然になる。「読み返してくどい」と感じた箇所だけ直す
- 表のセル内は「項目定義の核となる部分」を体言止め、続く動作や注意の説明をですます調で繋ぐと収まりが良い
  - 例: 「走査するディレクトリ名 (gems, packages, cdk など)。`.` を渡すとリポジトリルート直下を対象にします」

## コード変更

- 要求された箇所のみ変更する。リフォーマットや無関係なファイルの変更はしない
  - フォーマッタをグローバルに実行しない（例: `prettier --write .`）
- コミット・プッシュはユーザーから明示的に指示された場合のみ実行する
  - コミット前に差分を提示し、ユーザーの確認を得る
- 根本原因を見つける。一時的な修正は避ける
- ハック的な修正に感じたら、一度立ち止まってエレガントな解決策を検討する
- mainブランチとの差分を確認し「スタッフエンジニアはこれを承認するか？」と自問する

## コミットメッセージ

- `~/.claude/rules/commit.md` に従う
- 文脈から自明な情報は繰り返さない

## PR作成

- PR作成前に必ずリポジトリの PR テンプレートを確認する
  - `.github/PULL_REQUEST_TEMPLATE.md` または `.github/pull_request_template.md`、`docs/pull_request_template.md` を順に探す
  - テンプレートが存在する場合はそのセクション構成に従って本文を作成する
  - テンプレートが無い場合は「背景と目的」「主な変更内容」「動作確認方法」の 3 セクションをデフォルトで含める
- PRの説明文は簡潔に、何を変えたか・なぜ変えたかに焦点を当てる
  - 文体は「日本語ドキュメントの文体」に準拠する（ですます調）
  - 読み手はチャット履歴を見ていない前提で書く。会話で使った内部用語（「モードA」「pilot」「share リンク」などのその場限りの呼称）を持ち込まない。読み手が初見でも自己完結する文章にする
  - コードを見れば分かることは書かない。`git diff` で読み取れる「ファイルを追加した」「メソッド名を変えた」を列挙しない。設計判断の理由・前提・トレードオフなど、コードから読み取れない情報を優先する
  - 「merge したら振る舞いがどう変わるか」を中心に書く。ユーザー・他チーム・運用者から見た変化を伝える。実装詳細を網羅的に書く必要はない
  - 「何を自動化するか／しないか」「何を変えるか／変えないか」など対比構造があるときは表で明示する
  - 設計判断は「なぜそれを選んだか」を1-2文で添える。代替案を全部並べる必要はないが、自明でない選択には理由を残す
  - 動作確認方法は具体的なチェック項目で書く（「動作確認する」ではなく「Xをやって Y が起きることを確認する」）
- チケット/issue番号が提供されていない場合は必ず確認する
- PRはデフォルトでdraftとして作成する

## コマンド出力の分析

- 分析が必要なコマンドは出力を一時作業ディレクトリに保存する（例: `command 2>&1 | tee ~/.cache/claude-scratch/output.log`）
- 保存したファイルを `Read` ツールで分析する。同じコマンドをパイプを変えて再実行しない
  - 切り分け: `tail` や `Bash` で末尾やサマリを確認する
  - 深掘り: `Read` の `offset` / `limit` で該当箇所の前後を読む

## 進め方

- 3ステップ以上 or アーキテクチャに関わるタスクはPlanモードで開始する
- 途中でうまくいかなくなったら、無理に進めずすぐに立ち止まって再計画する
- リサーチ・調査・並列分析はサブエージェントに委譲し、1エージェント1タスクを守る
- ユーザーから繰り返し同じ指摘を受けるパターンは memory に feedback として記録する
- 言われなくても、失敗しているCIテストを修正しに行く
