# アップグレード / マイグレーション計画スキル

ユーザーの要求に基づいて、以下の構造でマイグレーション計画を作成する。
計画は `.claude/plans/` ディレクトリに Markdown ファイルとして保存する。

## 計画作成の手順

1. **調査**: 対象のコードベースを読み、現在の状態（バージョン、使用箇所、依存関係）を把握する
2. **影響範囲の特定**: 変更が必要なファイル・モジュールをすべて洗い出す
3. **計画の作成**: 以下のテンプレートに従って計画を書く
4. **ユーザーに確認**: 計画をレビューしてもらい、承認を得てから実装に進む

## 計画テンプレート

```markdown
# [移行元] → [移行先] マイグレーション計画

## Overview

移行の目的を1〜3文で説明する。なぜこの移行が必要か、完了後に何が改善されるかを書く。

## 前提条件

移行を開始する前に満たすべき条件をリストする。

- ランタイムやツールのバージョン要件
- 事前に完了しておくべき作業
- 必要な権限やアクセス

## 影響範囲

| ファイル | モジュール | 用途 |
|---------|----------|------|
| `path/to/file` | module-name | 何に使っているか |

## Steps

各ステップは1つの明確な目的を持ち、1コミットに対応する。
ステップの順序は、どの時点でもコードが動作する状態を保つように設計する。

### Step 1: [変更内容の要約]

**パス**: `対象ファイルのパス`

何をするか、なぜこの順序かを説明する。

before/after のコードを示す:

\```language
// Before
既存のコード

// After
変更後のコード
\```

**検証**:

- [ ] `ビルドコマンド` が成功する
- [ ] `テストコマンド` が通過する
- [ ] 変更の意図通りに動作することを確認する具体的な方法

### Step 2: ...

（Step を必要なだけ繰り返す）

## Rollback Plan

移行が途中で失敗した場合の復旧手順を書く。

- 各ステップは独立したコミットなので、`git revert` で個別に戻せる
- 特別な注意が必要な点（データマイグレーション、外部サービスへの影響など）があれば明記する
- 完全にロールバックする場合の手順

## Commit Strategy

コミットの構成と順序を示す。各コミットの type と subject を書く。

1. `type: subject` — 何をするか
2. `type: subject` — 何をするか
3. ...

**注意事項**:
- コミット固有の注意点（フォーマット変更による大きな差分、等）
```

## 計画作成時のルール

- **具体的に書く**: ファイルパス、コマンド、コード例を必ず含める。抽象的な指示は避ける
- **before/after を示す**: コード変更は変更前後を並べて示し、何が変わるか一目でわかるようにする
- **各ステップに検証を付ける**: ステップごとに、その変更が正しく動作することを確認する具体的なコマンドやチェック項目を書く
- **動作する状態を保つ**: どのコミットの時点でもビルド・テストが通る順序で設計する
- **実行前に前提を検証する**: 各ステップの実行前に、前提条件が実際に満たされているか確認する（例: exportが変更済みか、依存がインストール済みか）
- **影響範囲を明示する**: 変更対象のファイルをテーブルで一覧化し、漏れを防ぐ
- **バージョンを明記する**: 現在のバージョンと移行先のバージョンを具体的に書く。「最新」ではなく実際のバージョン番号を調べて記載する
